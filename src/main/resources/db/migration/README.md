# DB 마이그레이션 (Flyway)

운영(prod) 배포 시 Flyway가 이 폴더의 마이그레이션을 RDS에 자동 적용한다.

## 파일명 규칙 (타임스탬프 버전)

`V{yyyyMMdd}_{HHmm}__{설명}.sql`

예) `V20260718_1430__add_notification_table.sql`

- **번호를 직접 매기지 않고 만든 시각을 쓴다.** 여러 명이 각자 마이그레이션을 만들어도
  버전이 겹치지 않아 merge 충돌이 나지 않는다. (예전 순번 방식 V3, V4… 는 팀 작업 시 충돌이 잦아 폐기)
- Flyway는 `_` 를 버전 구분자로 처리하므로 `20260718_1430` → 버전 `20260718.1430` 으로 인식되어
  항상 시간순으로 정렬·적용된다.
- 언더바는 버전과 설명 사이 **2개(`__`)**, 버전 안에서는 **1개(`_`)**.

## 기존 이력 (건드리지 말 것)
- `V1` = 기존 RDS 전체를 baseline으로 마킹 (파일 없음, 적용 완료)
- `V2__rename_comment_to_comments.sql` = 적용 완료
- 위 둘은 **이미 배포되어 절대 수정하지 않는다.** 새 변경은 새 타임스탬프 파일로 추가한다.

## SQL 작성은 손으로 하지 말 것 (JPA Buddy)
엔티티를 바꾼 뒤 IntelliJ **JPA Buddy** 플러그인으로 마이그레이션을 자동 생성한다.
엔티티 우클릭 → *JPA Buddy* → *Generate Flyway Migration* → 현재 DB와 diff 하여 SQL 파일 생성.
파일명만 위 타임스탬프 규칙으로 맞추면 된다. (플러그인 없이 손으로 써도 되지만 권장하지 않음)

## 언제 작성하나
- 엔티티를 추가/변경해 **공용 DB 스키마가 바뀔 때**, 해당 PR에 마이그레이션 파일을 함께 포함한다.
- 로직만 바뀌고 스키마가 그대로면 **불필요**하다.
- 로컬은 `ddl-auto=update`로 동작해 로컬 개발 자체엔 이 파일이 필수는 아니지만,
  **운영 반영을 위해 반드시 작성**해야 한다. (미작성 시 배포에서 validate 실패 → 컨테이너 크래시)

## 주의
- 한 번 배포·적용된 마이그레이션 파일은 **수정하지 않는다.** 변경이 필요하면 새 파일을 추가한다.
- MySQL 예약어(`comment`, `order`, `group` 등) 테이블/컬럼명은 백틱으로 감싼다. (예: `` `comment` ``)
